Key ML Concepts: Gradient Descent, Hyperparameters & Cross-Validation

How a model actually learns — and how you tune and evaluate it properly.

1. Gradient Descent — How a Model Learns

When a model trains, it needs to find the best internal settings (called parameters) that minimize its mistakes. Gradient Descent is the algorithm that does this.

Here's the core idea in three steps:

Step 1 — Measure the error (Loss)
The model makes a prediction. We compare it to the correct answer and calculate how wrong it was. This "wrongness" score is called the loss (or cost). Lower loss = better model.
Step 2 — Calculate the gradient
A gradient tells you the direction and steepness of the loss curve — like knowing which way is "downhill." Mathematically it's a derivative: how much does the loss change if you nudge each parameter slightly?
Step 3 — Update parameters
Move each parameter a small step in the direction that reduces the loss. Repeat thousands of times until the loss stops decreasing.
Analogy — Hiking Down a Mountain Blindfolded
Imagine you're dropped blindfolded onto a mountain and need to find the lowest valley. You can't see far, but you can feel the slope under your feet. You take a step in whichever direction feels most downhill. Repeat. Eventually you reach a low point. That's gradient descent — the "mountain" is the loss landscape, the "valley" is the minimum loss.

Each step moves parameters toward the minimum loss

Three Variants of Gradient Descent

Batch Gradient Descent (BGD)
Calculates the gradient using the entire training dataset before taking one step.
Pro: Smooth, stable convergence. Con: Very slow on large datasets — one step requires processing all data.
Stochastic Gradient Descent (SGD)
Takes a step after each single training example.
Pro: Very fast updates, can escape local minima. Con: Noisy — the loss bounces around rather than descending smoothly.
Mini-batch Gradient Descent (MBGD)
Takes a step after each small batch (e.g. 32 or 64 examples).
Pro: Best of both worlds — faster than BGD, smoother than SGD. This is the standard in practice.
⚠ Exam Alert
Know all three variants. The practical standard is Mini-batch Gradient Descent. BGD uses all data; SGD uses one sample; MBGD uses a small batch. "Batch size" is a hyperparameter you choose.

2. Parameters vs. Hyperparameters

These two terms are easy to confuse. The distinction is about who sets them.

Parameters
  • Learned automatically by the model during training
  • You never set these manually
  • Change with every training run
Examples: weights and biases in a neural network, coefficients in linear regression
Hyperparameters
  • Set by you before training begins
  • Control how training happens
  • Not learned from data
Examples: learning rate, number of layers, batch size, number of trees in a forest, max tree depth
Analogy
Think of baking bread. Hyperparameters are the settings you choose before baking: oven temperature (200°C), baking time (30 min), flour amount. Parameters are what the bread becomes: its texture, crust, rise — determined by the baking process itself, not chosen in advance.

The Learning Rate — The Most Important Hyperparameter

The learning rate controls how big each step is during gradient descent. It's often the most critical hyperparameter to tune:

Too high
Steps are too large — you overshoot the minimum and bounce around, or diverge entirely. Loss goes up instead of down.
Too low
Steps are tiny — training takes forever, or gets stuck in a local minimum.
Just right
Descends steadily toward the minimum in a reasonable number of steps.

3. Hyperparameter Search Methods

Finding the best hyperparameters is itself a search problem. You can't learn them from data — so how do you find good values?

① Grid Search
Define a grid of values to try for each hyperparameter. Train a model for every combination. Pick the best.

Example: Try learning rates [0.001, 0.01, 0.1] × batch sizes [16, 32, 64] = 9 combinations, train all 9.

Pro: Exhaustive — guaranteed to find the best within the grid. Con: Exponentially slow as you add more hyperparameters.
② Random Search
Randomly sample hyperparameter combinations and train on each. Stop after a budget of trials.

Pro: Much faster than grid search and often finds good results sooner (random samples cover the space more efficiently). Con: Not exhaustive — might miss the best combination.
③ Heuristic / Intelligent Search
Use domain knowledge or evolutionary algorithms (genetic algorithms, etc.) to guide the search toward promising regions. Smarter than random but requires more setup.
④ Bayesian Search (Bayesian Optimization)
Builds a probabilistic model of which hyperparameter combinations are likely to work well, based on results so far. Uses that model to pick the most promising next combination to try.

Pro: Far more efficient than grid or random — learns from previous trials to focus on good regions. Con: More complex to implement. This is the state-of-the-art in practice.
⚠ Exam Alert
Know all four methods in order of sophistication: Grid Search (exhaustive, slow) → Random Search (faster, good coverage) → Heuristic Search (domain-guided) → Bayesian Search (learns from previous trials, most efficient). Bayesian is most efficient; Grid is simplest.

4. Cross-Validation

Remember: you split data into training and test sets to get an honest performance measure. But there's a problem — what if you happened to get a "lucky" or "unlucky" split? Your test score might not be reliable.

Cross-validation solves this by evaluating the model on multiple different splits and averaging the results.

How it works: Split the data into k equal parts ("folds"). Train on k−1 folds, test on the remaining 1. Rotate which fold is the test set. Repeat k times. Average the k scores.

Why it's better than a single split: Every data point gets used for testing exactly once. The averaged score is a much more reliable estimate of true performance.

Common values of k
k=5 and k=10 are the most common choices. Larger k = more reliable estimate but more computation. k = number of samples is called Leave-One-Out Cross-Validation (LOOCV) — the most thorough but very slow.
⚠ Exam Alert
Cross-validation purpose: get a more reliable performance estimate by testing on multiple different splits and averaging. K-Fold CV splits data into k parts, trains k times (each time using a different fold as test), then averages. It does not replace the final test set — it's used during model development to compare models or tune hyperparameters.
